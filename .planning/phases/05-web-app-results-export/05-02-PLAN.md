```typescript
import type { ExtractionErrorCode } from "@/lib/docpipe";

export type ResultErrorState = {
  code: ExtractionErrorCode | "CLIENT";
  title: string;
  message: string;
  details?: string[];
};

export function toExtractionErrorState(error: unknown): ResultErrorState;
```

Map the main error codes to these exact user-facing titles and messages:
- `INVALID_API_KEY`
  - title: `API key rejected`
  - message: `Your Anthropic API key was rejected by Anthropic. Check the key and retry.`
- `UNSUPPORTED_FILE_TYPE`
  - title: `Unsupported file format`
  - message: `Use a PDF, PNG, or JPG document and try again.`
- `EXTRACTION_FAILED`
  - title: `Extraction failed`
  - message: `The document could not be extracted into a valid result. Retry the same file or choose a clearer scan.`
- `VALIDATION_FAILED`
  - title: `Schema validation failed`
  - message: `The model could not satisfy the schema after the retry limit. Simplify the schema or adjust the document.`
  - details: use `validationErrors` from the extraction error when present
- `RATE_LIMITED`
  - title: `Rate limited`
  - message: `Anthropic rate-limited this request. Wait a moment, then retry.`
- fallback / client-side file-reader or schema parser errors
  - code: `CLIENT`
  - title: `Client-side error`
  - message: use the thrown message when available, otherwise `Something went wrong in the browser before extraction completed.`

Refactor `apps/web/src/components/docpipe/docpipe-workspace.tsx` so it stores:
```typescript
const [result, setResult] = useState<ExtractionResult<Record<string, unknown>> | null>(null);
```

Remove the separate `resultJson` and `overallConfidence` state variables. On success, store the full `result` object from `extract()`. On failure, store `toExtractionErrorState(error)` and clear `result`. Clear `resultError` when the selected file, template choice, API key, or custom schema source changes.

Pass `result` and `resultError` into `ResultsPreview`. Do not keep a second JSON-only source of truth in the workspace.
